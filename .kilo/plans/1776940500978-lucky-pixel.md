# Plan: Admin + Client Support Access System

**Spec:** `docs/superpowers/specs/2026-04-23-admin-client-access-design.md`  
**Status:** Ready to implement

---

## What We're Building

A platform admin system allowing Leadder team members to:
1. Log in as a normal user and be routed to `/admin` dashboard
2. Browse all client businesses
3. Impersonate a client's portal (only if client has opted in)
4. Create new admin users
5. Exit impersonation back to admin dashboard

**File: `scripts/clean/006_platform_admins.sql`**
```sql
CREATE TABLE public.platform_admins (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  created_by  UUID REFERENCES auth.users(id),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id)
);
```

### 3. Supabase Server Helpers (`lib/supabase/server.ts`)

Add two new exported async functions:

**`createServiceClient()`** — uses `SUPABASE_SERVICE_ROLE_KEY` (env var, not public). Creates a service-role Supabase client that bypasses RLS.

**`isPlatformAdmin(userId: string): Promise<boolean>`** — queries `platform_admins` using the service client; returns `true` if a row exists for `userId`.

### 4. Middleware Update (`lib/supabase/proxy.ts`)

Extend `updateSession` to also guard `/admin` routes:
- If `request.nextUrl.pathname.startsWith('/admin')` and `!user` → redirect to `/auth/login`
- (Admin *identity* check is done in `app/admin/layout.tsx`, not here)

### 5. New API Routes

#### `app/api/admin/impersonate/route.ts` (POST)
1. Get current user via `getUser()`; if none → 401
2. Call `isPlatformAdmin(user.id)`; if false → 403
3. Parse `{ businessId }` from request body
4. Query `business_settings` (service role) for `support_access_enabled`; if false → 403 with message
5. Set HttpOnly cookie `leadder_impersonating_business_id=<businessId>` (SameSite=Lax, Path=/)
6. Return 200 JSON `{ ok: true }`

#### `app/api/admin/exit/route.ts` (POST)
1. Delete cookie `leadder_impersonating_business_id`
2. Return redirect to `/admin` (or JSON `{ ok: true }` — redirect handled client side)

#### `app/api/admin/create-user/route.ts` (POST)
1. Get current user; if none → 401
2. `isPlatformAdmin(user.id)`; if false → 403
3. Parse `{ email, password }` from body
4. Use service role `supabase.auth.admin.createUser({ email, password, email_confirm: true })`
5. Insert row into `platform_admins(user_id, created_by)` using the new user's id and caller's id
6. Return 200 JSON `{ ok: true }`

### 10. Portal Layout Update (`app/portal/layout.tsx`)

Modify the existing layout to:

1. **Read impersonation cookie** (`leadder_impersonating_business_id`) from `cookies()`
2. **If cookie present:**
   a. Verify `isPlatformAdmin(user.id)` (server-side, NOT from cookie)  
   b. Fetch `business_settings.support_access_enabled` for cookie's business ID (service role)  
   c. If `support_access_enabled=false`: clear cookie + redirect to `/admin`  
   d. Resolve business via service role: `.from('businesses').eq('id', cookieBusinessId)`  
   e. Pass `isImpersonating=true` + `businessName` down
3. **If cookie absent (normal flow):**
   a. Check if user is a platform admin: if yes → redirect to `/admin`  
   b. Otherwise current logic: fetch business by `owner_id`, redirect to `/auth/sign-up` if no business
4. Render `<ImpersonationBanner>` when impersonating (inside the layout, above sidebar)

## Key Design Decisions (from spec)

- **No RLS on `platform_admins`** — service role only
- **Cookie is HttpOnly, SameSite=Lax** — not readable by JS
- **Revocation is immediate** — checked on every portal layout render
- **All security enforcement is server-side** — banner is purely informational
- **Admin identity gate** is in `app/admin/layout.tsx`, not middleware

---

## Out of Scope (per spec)

- Audit logging
- Email notifications on support access toggle
- Per-admin permission levels
